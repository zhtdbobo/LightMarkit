# 提交日志规范

本项目提交信息统一使用 Conventional Commit 风格，标题格式如下：

```text
type(scope): 中文描述
```

## 标题规则

- `type` 使用英文小写，必须从下方列表选择：
  - `feat`：新增功能
  - `fix`：问题修复
  - `chore`：维护、配置、依赖、构建辅助
  - `docs`：文档
  - `test`：测试
  - `refactor`：重构，不改变行为
  - `style`：格式、样式、排版
  - `perf`：性能优化
  - `build`：构建系统或打包配置
  - `ci`：持续集成
  - `revert`：回滚提交
- `scope` 使用英文小写，描述影响范围。多个单词用短横线连接。
- 冒号后必须是中文描述；技术名词可以保留英文，例如 `Markdown`、`PDF`、`Tauri`、`NSIS`。
- 标题尽量控制在一行内，优先简洁说明“做了什么”。
- 标题末尾不加句号。

## 推荐 scope

- `app`：应用整体流程
- `editor`：编辑器
- `preview`：预览
- `export`：导出流程
- `pdf`：PDF 导出
- `window`：窗口、标题栏、关闭/最小化/最大化
- `installer`：安装包
- `security`：安全与输入处理
- `scaffold`：项目骨架
- `import-export`：导入导出整体能力
- `docs`：文档
- `test`：测试
- `deps`：依赖

## 正文规则

- 简单修改只写标题，不写正文。
- 需要正文时，正文也使用中文；技术名词可以保留英文。
- 正文用于说明原因、影响范围、验证方式，不重复标题。
- 避免提交信息里混入大段英文说明。

## 示例

推荐：

```text
fix(app): 支持旧编码 Markdown 文件并修复窗口关闭
feat(export): 改进 Markdown 预览和导出流程
fix(installer): 规范化 NSIS 安装目录
chore(pdf): 使用 headless_chrome 默认选项实现 PDF 导出并更新配置
```

不推荐：

```text
fix: support legacy markdown files and window close
功能：完善 Markdown 打开与预览体验
fix(pdf):pdf仍然会有大片空白问题
```

## 修改已推送历史

已推送的提交信息原则上不重写。确实需要统一历史时：

- 先确认没有其他人基于旧历史继续开发。
- 使用 `--force-with-lease`，不要使用普通 `--force`。
- 重写后提醒其他机器同步新的远端历史。
