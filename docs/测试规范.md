# LightMarkit 测试规范

> 每一个功能开发完成后都必须编写测试用例进行验证

## 测试框架

- **测试运行器**：Vitest
- **React 测试**：@testing-library/react
- **用户交互**：@testing-library/user-event
- **DOM 断言**：@testing-library/jest-dom
- **测试环境**：happy-dom

## 测试命令

```bash
# 开发模式运行测试（监视模式）
pnpm test

# 运行所有测试并退出
pnpm test:run

# 运行测试并打开 UI 界面
pnpm test:ui

# 运行测试并生成覆盖率报告
pnpm test:coverage
```

## 测试编写规范

### 1. 组件测试

每个 React 组件必须包含以下测试：

```typescript
import { describe, it, expect } from 'vitest'
import { render, screen } from '@testing-library/react'
import userEvent from '@testing-library/user-event'
import MyComponent from './MyComponent'

describe('MyComponent', () => {
  // ✅ 必需：基础渲染测试
  it('应该正确渲染', () => {
    render(<MyComponent />)
    expect(screen.getByText('预期文本')).toBeInTheDocument()
  })

  // ✅ 必需：Props 测试
  it('应该正确处理 props', () => {
    render(<MyComponent title="测试标题" />)
    expect(screen.getByText('测试标题')).toBeInTheDocument()
  })

  // ✅ 必需：用户交互测试
  it('应该响应用户交互', async () => {
    const user = userEvent.setup()
    const handleClick = vi.fn()
    render(<MyComponent onClick={handleClick} />)
    
    await user.click(screen.getByRole('button'))
    expect(handleClick).toHaveBeenCalledTimes(1)
  })

  // ✅ 必需：状态变化测试
  it('应该正确更新状态', async () => {
    const user = userEvent.setup()
    render(<MyComponent />)
    
    const input = screen.getByRole('textbox')
    await user.type(input, 'Hello')
    
    expect(input).toHaveValue('Hello')
  })
})
```

### 2. Hook 测试

自定义 Hook 必须使用 `renderHook` 测试：

```typescript
import { describe, it, expect } from 'vitest'
import { renderHook, waitFor } from '@testing-library/react'
import { useAutoSave } from './useAutoSave'

describe('useAutoSave', () => {
  it('应该在内容变化后自动保存', async () => {
    const saveFn = vi.fn()
    const { result } = renderHook(() => useAutoSave(saveFn, 500))
    
    result.current.trigger('新内容')
    
    await waitFor(() => {
      expect(saveFn).toHaveBeenCalledWith('新内容')
    }, { timeout: 600 })
  })
})
```

### 3. 工具函数测试

纯函数测试覆盖所有边界情况：

```typescript
import { describe, it, expect } from 'vitest'
import { parseMarkdown } from './markdown'

describe('parseMarkdown', () => {
  it('应该正确解析标题', () => {
    const result = parseMarkdown('# 标题')
    expect(result).toContain('<h1>标题</h1>')
  })

  it('应该处理空输入', () => {
    const result = parseMarkdown('')
    expect(result).toBe('')
  })

  it('应该转义 HTML 标签', () => {
    const result = parseMarkdown('<script>alert("XSS")</script>')
    expect(result).not.toContain('<script>')
  })
})
```

### 4. Tauri Command Mock

测试涉及 Tauri API 的组件时，必须 Mock：

```typescript
import { vi } from 'vitest'

vi.mock('@tauri-apps/api/core', () => ({
  invoke: vi.fn(),
}))

describe('FileManager', () => {
  it('应该调用文件读取命令', async () => {
    const { invoke } = await import('@tauri-apps/api/core')
    const mockInvoke = vi.mocked(invoke)
    mockInvoke.mockResolvedValue('文件内容')

    render(<FileManager />)
    // ... 测试逻辑
  })
})
```

## 测试最佳实践

### ✅ 推荐做法

```typescript
// ✅ 使用 screen 查询
expect(screen.getByText('按钮')).toBeInTheDocument()

// ✅ 使用 user-event 模拟真实交互
const user = userEvent.setup()
await user.click(button)
await user.type(input, 'text')

// ✅ 使用语义化查询
screen.getByRole('button', { name: '提交' })
screen.getByLabelText('用户名')

// ✅ 等待异步更新
await waitFor(() => {
  expect(screen.getByText('加载完成')).toBeInTheDocument()
})

// ✅ 测试可访问性
expect(screen.getByRole('button')).toHaveAttribute('aria-label', '关闭')
```

### ❌ 避免做法

```typescript
// ❌ 不要使用 container 查询
const { container } = render(<Component />)
container.querySelector('.class-name') // 脆弱

// ❌ 不要直接使用 fireEvent（除非必须）
fireEvent.click(button) // 不够真实

// ❌ 不要依赖实现细节
expect(component.state.value).toBe('test') // 测试实现而非行为

// ❌ 不要使用固定延时
await new Promise(resolve => setTimeout(resolve, 1000)) // 不可靠
```

## 示例：完整的功能测试

以 M1.2 任务（CodeMirror 编辑器集成）为例：

```typescript
// src/components/Editor/Editor.test.tsx
import { describe, it, expect } from 'vitest'
import { render, screen, waitFor } from '@testing-library/react'
import userEvent from '@testing-library/user-event'
import Editor from './Editor'

describe('Editor', () => {
  describe('基础功能', () => {
    it('应该渲染 CodeMirror 编辑器', () => {
      render(<Editor />)
      expect(screen.getByRole('textbox')).toBeInTheDocument()
    })

    it('应该显示初始内容', () => {
      render(<Editor initialValue="# Hello World" />)
      expect(screen.getByText('# Hello World')).toBeInTheDocument()
    })
  })

  describe('Markdown 语法高亮', () => {
    it('应该高亮标题语法', () => {
      render(<Editor />)
      // 验证语法高亮类名
      const heading = screen.getByText('#')
      expect(heading).toHaveClass('cm-heading')
    })

    it('应该高亮粗体语法', () => {
      render(<Editor initialValue="**bold**" />)
      const bold = screen.getByText('**')
      expect(bold).toHaveClass('cm-strong')
    })
  })

  describe('用户交互', () => {
    it('应该在用户输入时更新内容', async () => {
      const user = userEvent.setup()
      const handleChange = vi.fn()
      render(<Editor onChange={handleChange} />)

      const editor = screen.getByRole('textbox')
      await user.type(editor, '# Test')

      await waitFor(() => {
        expect(handleChange).toHaveBeenCalledWith('# Test')
      })
    })

    it('应该支持多行输入', async () => {
      const user = userEvent.setup()
      render(<Editor />)

      const editor = screen.getByRole('textbox')
      await user.type(editor, 'Line 1{Enter}Line 2')

      expect(editor).toHaveValue('Line 1\nLine 2')
    })
  })

  describe('错误处理', () => {
    it('应该处理空内容', () => {
      render(<Editor initialValue="" />)
      expect(screen.getByRole('textbox')).toHaveValue('')
    })

    it('应该处理超长文本', () => {
      const longText = 'a'.repeat(10000)
      render(<Editor initialValue={longText} />)
      expect(screen.getByRole('textbox')).toBeInTheDocument()
    })
  })
})
```

## 总结

### 开发流程检查清单

每完成一个功能时，确保：

- [ ] 编写了单元测试
- [ ] 测试覆盖了核心逻辑
- [ ] 所有测试通过 (`pnpm test:run`)
- [ ] 没有控制台错误或警告
- [ ] 代码通过 lint 检查 (`pnpm lint`)
- [ ] 格式化代码 (`pnpm format`)
- [ ] 更新了相关文档

### 问题排查

**测试运行失败？**
1. 检查是否安装了所有依赖：`pnpm install`
2. 检查是否正确 Mock 了 Tauri API
3. 检查异步操作是否使用了 `waitFor`
4. 运行 `pnpm test:ui` 使用 UI 调试

**测试覆盖率不足？**
1. 运行 `pnpm test:coverage` 查看详细报告
2. 补充边界条件测试
3. 添加错误处理测试
4. 测试用户交互场景

---

**记住**：测试不是负担，而是保证代码质量和重构信心的基础。养成先写测试的习惯，长期收益巨大。
